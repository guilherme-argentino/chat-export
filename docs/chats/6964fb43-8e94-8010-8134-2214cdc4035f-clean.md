## 6. O Conceito Central: Control Plane
**Assistente:** O conceito é **Control Plane**. Mais precisamente: Um Control Plane *multi-tenant*, *spec-driven*, que governa múltiplos *runtimes* autônomos.

### 1) O nome "correto" do conceito (sem marketing)
Em engenharia de software e sistemas distribuídos, isso é a combinação de:
*   Control Plane vs. Data Plane
*   Governança declarativa
*   *Spec-driven systems*
*   *Policy-based execution*

Você está redesenhando o modelo de sistemas como **Kubernetes, Stripe, AWS Control Tower, Auth0 e GitHub**, mas aplicado a SaaS + Produto + Desenvolvimento + IA.

### 2) Como o conceito aparece no projeto
*   **Kernel:** Control Plane do negócio. Governa tenants, identidade, direitos (*entitlements*) e runtimes.
*   **Runtime (Curso, Lab, App):** Data Plane. Executa o trabalho, mas não decide políticas.
*   **IA:** Outro runtime. Governado por *specs*, produz artefatos submetidos a *quality gates*.
*   **On-prem:** Apenas muda onde o *data plane* roda; o *control plane* continua central.

### 3) Por que isso resolve o seu "bloqueio histórico"?
Você ficava preso no *core* porque tentava **implementar** um control plane quando deveria estar **definindo e governando** um control plane.

### 4) Mudança de Postura
*   **Antes:** Você escrevia código, decidia fluxos, resolvia exceções. (Perfil *Builder*).
*   **Agora:** Você escreve *specs*, define políticas, aceita/rejeita artefatos e governa agentes. (Perfil *Governor*).

> **"Se você está discutindo implementação, você já perdeu o controle. Implementação pertence ao Data Plane. Você opera no Control Plane."**

### 5) Nome Técnico sugerido
*   *Spec-Driven Multi-Tenant Control Plane for Governed Runtimes* (ou simplesmente **Unified Control Plane**).
